# Formal-to-Implementation Refinement Audit

Audited: 2026-04-30

This document answers the practical question behind the Tamarin files: are the
facts proved in `docs/formal` actually the facts the implementation enforces?

Short answer: after the fixes noted below, the formal claims line up with the
implemented protocol at their stated abstraction level. The models are still
symbolic and intentionally smaller than the system: they prove provenance,
secrecy, signature separation, AAD binding, authorization gates, and monotonic
state transitions, while tests and code review cover SQL shape, browser/OPFS
behavior, deployment hardening, performance, and UI flows.

## Changes Made During This Audit

- `crypto-ops.ts`: domain-table import now rejects rows when sender permission
  resolution fails. Previously `resolveSenderPermissions(...) === null` skipped
  table permission checks; that did not refine the
  `accepted_delta_requires_authorized_sender` lemma.
- `PermissionEnforcementTest.cs`: added browser-backed coverage for an invalid
  ShareTarget credential producing `PERMISSION_SENDER_UNAUTHORIZED` and no
  open/shadow materialization.
- `GroupService.UpdateMemberRoleAsync`: role changes now require admin keys and
  re-sign the ShareTarget credential. Previously a role change could leave a
  stale AdminSignature over the old role.
- `05-pin-purge-authority.spthy`: comments/model variable names now match the
  implemented hash-based POST canonical string.

## Refinement Boundaries

- Cryptographic primitives are idealized in Tamarin. Correctness of sodium,
  WebCrypto, noble, BouncyCastle, and `crypto-core` is tested/trusted outside
  the symbolic model.
- Secret-key lemmas assume no memory disclosure or same-origin live process
  compromise. JS/WASM heap disclosure is out of scope.
- Random nonces are modeled as fresh atoms. The implementation uses 96-bit
  random nonces; tests catch regressions, while birthday-bound risk remains an
  operational scale question.
- Relay receive cursor monotonicity is a client-side property implemented by
  `HttpSyncTransport` plus `IReceiveCursorStore`; the relay still accepts any
  signed `since=N` query.
- Pin purge removes prior rows from the relay. It does not erase rows already
  fetched and stored locally by a client before the purge.
- Receiver `CanRead` is not a sync-layer materialization gate under the current
  full-local-snapshot policy.

## CryptoSync 03 - Delta Data Plane

| Formal claim | Implementation evidence | Verdict |
|---|---|---|

`DeltaEnvelopeTests.cs`, `PlaintextLeakProperty.cs`, Playwright/TestApp
encrypted delta cases, `PermissionEnforcementTest.cs`.

Modeled abstraction:
one row/group and one CEK per envelope. Current worker comments also state that
multi-group-per-envelope CEK resolution is future work.

| Formal claim | Implementation evidence | Verdict |
|---|---|---|

`relay-integration.php` admin/non-admin pin cases and
`HttpSyncTransportLiveRelayTests.SeedReseed_GcSignalDrivesAdminCompaction_OperationalRoundTrip`.

## VFS PRF Slot Encryption

| Formal claim | Implementation evidence | Verdict |
|---|---|---|
| `key_secrecy` | Per-path keys live in `key-registry.ts` and are never written to OPFS by the VFS. | Aligned, assuming no memory disclosure. |
| `encrypted_slot_secrecy_unless_plain_exported` | Encrypted xWrite seals full 4096-byte slots as ChaCha20-Poly1305 ciphertext; plain export/rekey-to-plain intentionally releases plaintext bytes. | Aligned. |
| `encrypted_read_authenticity` | xRead decrypts only with the matching key and `buildPageAad(path, slotIndex)`; tag failure aborts the read. | Aligned. |
| `verify_key_match_sound` | `verifyEncryptionKey(path)` probes slot 0 with the registered key and slot-0 AAD. | Aligned. |
| `rekey_encrypted_to_plain_sound` | `rekeySlots` decrypts each encrypted source slot before writing 4096-byte plain pages. | Aligned. |
| `rekey_encrypted_to_encrypted_sound` | `rekeySlots` decrypts under source key and re-encrypts the same plaintext under target key and the same AAD. | Aligned. |
| `legacy_ciphertexts_not_read_as_v1` | AAD includes the `prf-vfs-v1|` prefix; tests cover wrong AAD/cross-version/cross-path rejection. | Aligned. |
| `nonce_never_reused` | Implementation uses random 96-bit nonces. Tests catch constant/reused/counter-reset regressions. | Symbolic claim is stronger than implementation; acceptable as standard nonce-freshness abstraction with operational birthday-bound caveat. |

Main executable coverage:
`vfs-prf/__tests__/envelope.test.ts`, `vfs-prf/__tests__/rekey.test.ts`,
`PrfVfsEnvelopeTests.cs`, `nonce-uniqueness.test.ts`.

## Current Verdict

No known formal claim in `docs/formal` is deliberately stronger than the
implementation without being called out above as a symbolic abstraction or
threat-model boundary. The most important refinement bug found by this audit
was the delta import null-permission fallback; that is now a hard rejection.
